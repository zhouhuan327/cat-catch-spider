## 2. 当前仓库的实现结构

这个仓库本质上是一个浏览器资源嗅探扩展，不是传统后端爬虫。核心能力来自三条链路叠加：

- 网络层抓取：监听浏览器请求和响应
- 页面运行时深抓：注入脚本 hook `fetch/XHR/JSON.parse/Worker/TypedArray` 等对象
- 下载与解析工具页：对 `m3u8/mpd/普通文件` 做专门处理

### 2.1 主要模块分工

#### 1) 后台层

文件：

- `js/background.js`

职责：

- 监听 `webRequest` / `webNavigation`
- 调用 `findMedia()` 对请求进行媒体识别
- 维护全局媒体缓存 `cacheData`
- 响应 popup / preview / downloader / content-script 的消息
- 管理脚本注入、快捷键、图标角标、自动下载和标签页状态

这是当前系统的中心控制器，也是耦合最高的文件。

#### 2) 全局基础层

文件：

- `js/init.js`
- `js/function.js`

职责：

- 初始化全局配置 `G`
- 从 `storage.sync` / `storage.session` / `storage.local` 恢复配置
- 提供模板替换、请求头改写、清理缓存、打开解析页、发送到本地等公共能力

`G` 和 `cacheData` 是整个仓库的两套核心共享状态。

#### 3) 页面桥接层

文件：

- `js/content-script.js`

职责：

- 操作页面内的 `video/audio`
- 处理播放、暂停、倍速、截图、PIP、全屏
- 接收页面脚本 `window.postMessage`
- 把页面内深抓到的媒体 URL、密钥、ffmpeg 请求转发给扩展后台

#### 4) 页面内注入脚本

文件：

- `catch-script/search.js`
- `catch-script/catch.js`
- `catch-script/recorder.js`
- `catch-script/recorder2.js`
- `catch-script/webrtc.js`

职责：

- `search.js`：深度搜索，hook 页面运行时，找 m3u8 / mpd / key / blob 线索
- `catch.js`：代理 `MediaSource` / `appendBuffer` / `endOfStream`，直接截媒体缓冲
- `recorder*.js` / `webrtc.js`：做录制和屏幕/WebRTC 捕获

这部分功能最强，但兼容性风险也最高。

#### 5) UI 和工具页

文件：

- `popup.html` + `js/popup.js`
- `preview.html` + `js/preview.js`
- `m3u8.html` + `js/m3u8.js`
- `downloader.html` + `js/downloader.js`
- `mpd.html` + `js/mpd.js`
- `options.html` + `js/options.js`

职责：

- `popup`：轻量资源面板
- `preview`：网格预览、筛选、批量操作
- `m3u8`：解析、选流、密钥处理、解密、合并下载
- `downloader`：普通文件下载、流式下载、ffmpeg 发送
- `mpd`：将 DASH 结构转成可操作列表并可导向 m3u8 流程
- `options`：扩展的所有配置项

## 3. 当前核心数据流

### 3.1 普通抓取链

1. 用户访问页面
2. `background.js` 监听请求头和响应头
3. `findMedia()` 根据 URL 后缀、`content-type`、`content-disposition`、正则规则做过滤
4. 命中后构建统一 `info` 对象
5. 写入 `cacheData[tabId]`
6. 通过消息推给 popup / preview

### 3.2 深度搜索链

1. 用户开启 `search.js`
2. 页面中注入 hook 脚本
3. 页面运行时一旦解析出 m3u8 / mpd / key / 媒体 URL，先 `postMessage`
4. `content-script.js` 接收消息并 `chrome.runtime.sendMessage`
5. `background.js` 再次统一走 `findMedia()`

这条链解决的是“网络层不明显、但页面 JS 内能拿到”的资源。

### 3.3 下载链

1. popup / preview 选择资源
2. 优先尝试把任务发给现有 `downloader.html`
3. 如果没有下载器页，则创建新下载器页
4. 对于 m3u8 / mpd，则跳到专门解析页
5. 解析页再决定：
   - 直接下载
   - 流式写入
   - 发到在线 ffmpeg
   - 使用 m3u8DL / invoke 协议

### 3.4 请求头链

下载和预览依赖 `setRequestHeaders()` / `setHeaders()` 动态写入 `declarativeNetRequest` session rules。

这意味着：

- 如果这块出错，很多需要 `referer/cookie/auth` 的资源会直接失效
- 这是后续改版时不能轻易动坏的关键基础设施

## 4. 本次更新的主要内容

从 `5bb9d44` 到 `2388f26` 的更新不是单点修复，而是围绕以下几个方向做了持续增强。

### 4.1 m3u8 工具页大幅重构

涉及文件：

- `js/m3u8.js`
- `m3u8.html`

这是本轮更新最重的部分，变化主要有：

#### 1) 新增 iframe FFmpeg 模式

- 新增配置项 `iframeFFmpeg`
- `m3u8` 页可直接在当前页面内嵌在线 ffmpeg
- 不再只能依赖单独打开远程 ffmpeg 标签页

影响：

- 交互更闭环
- 远程 ffmpeg 通信链路从“扩展 tab 消息”扩展为“iframe + postMessage”
- 后续如果要改造 ffmpeg 能力，这一层已经开始分叉

#### 2) 新增主从任务合并模式

- 增加 `isMaster` 参数
- 使用 `BroadcastChannel('m3u8Channel')`
- 多个 m3u8 合并到 ffmpeg 时，主任务负责聚合，子任务通过频道回传数据

影响：

- 解决多个 m3u8 合并时的协同问题
- 但把任务协调逻辑分散到了 `popup/preview/function/m3u8` 多处

#### 3) m3u8 页面 UI 重做

更新点包括：

- 原始 m3u8、切片列表、在线 ffmpeg、视频预览切换为多面板视图
- 原先纯文本切片输出改为交互式 `mediaList`
- 切片支持逐条勾选、复制、进度展示、错误态
- 更多按钮折叠为弹出面板

影响：

- 可用性明显提升
- `js/m3u8.js` 现在同时承担了解析逻辑、下载逻辑、列表渲染、UI 状态机四种职责，复杂度继续升高

#### 4) 自动合并与进度处理增强

- 自动下载时改为调用 `autoMerge()`
- 直播记录由“记录最后一个 URL”改为“记录最后一个 `sn`”
- 估算文件大小采样从 `3` 片提高到 `5` 片

影响：

- 自动化更稳定
- 直播断点判断更合理

---

### 4.2 downloader 页支持 iframe FFmpeg

涉及文件：

- `js/downloader.js`
- `downloader.html`

更新点：

- 下载页新增 `#iframeBox`
- `sendFile()` 支持往内嵌 ffmpeg iframe 发送 Blob
- ffmpeg 返回状态从单一 `ok`，扩展为 `ok/done` 等阶段性状态
- 自动关闭延迟和关闭逻辑更统一

影响：

- downloader 不再只依赖独立 ffmpeg 标签页
- 普通下载链和 m3u8 下载链在 ffmpeg 通道上开始收敛

---

### 4.3 preview 页增强：状态记忆、源页面 DOM、框选体验

涉及文件：

- `js/preview.js`
- `css/preview.css`

更新点：

#### 1) 预览筛选与排序记忆

- 扩展名筛选状态记忆到 `sessionStorage`
- 类型筛选状态记忆到 `sessionStorage`
- 排序字段和顺序记忆到 `sessionStorage`

影响：

- 多页预览体验更连贯
- 也意味着 preview 页面状态开始不只依赖 `G`

#### 2) 支持获取源页面 DOM

- 新增 `sourcePageDOM`
- 若 `G.getHtmlDOM` 打开，则主动从源 tab 获取 HTML DOM
- `trimData()` 里通过 getter 把 `pageDOM` 挂回媒体对象

影响：

- 模板系统里的 `${xxx|find:selector}` 在 preview 页可用
- 后续如果要做“按页面内容命名文件”的能力，这已经打通

#### 3) 框选支持 Ctrl 保留已有选中

- 新增 `ctrlKeyPressed`
- 框选时如果按住 Ctrl，不会把其他已选中项取消

影响：

- 批量操作体验更合理

#### 4) 合并下载支持主从 m3u8 任务

- preview 发起 m3u8 合并时，会给第一个任务打 `isMaster`

---

### 4.4 popup 页与交互微调

涉及文件：

- `js/popup.js`
- `popup.html`
- `css/popup.css`

更新点：

- m3u8 合并任务同样改成主从模式
- 弹出新页时尽量插到当前 tab 后面
- `G.popup` 自动弹出时改成 `window.close()`，不再走 `closeTab()`
- 重排了部分按钮位置
- `duplicateFilenames` 从 filter 区域移到 features
- `openFilter`、展开/折叠相关按钮默认隐藏
- 正则图标类名从 `favicon regex` 改为独立 `regex`

影响：

- popup 更偏“精简操作台”
- 一部分旧交互入口被弱化了

### 4.5 options 页支持更强的大小过滤表达式

涉及文件：

- `js/options.js`
- `options.html`
- `css/options.css`
- `js/init.js`
- `js/background.js`

这是第二个非常重要的更新方向。

#### 1) 大小过滤不再只是“阈值 KB”

之前：

- `size` 本质上只支持“最小值过滤”

现在：

- 新增 `operator`
- 新增 `unit`
- 支持：
  - `>`
  - `<`
  - `>=`
  - `<=`
  - `=`
  - `!=`
  - `~` 区间

#### 2) `background.js` 增加统一 `operatorCheck()`

- `CheckExtension()`
- `CheckType()`

现在都走统一操作符判断逻辑

#### 3) `init.js` 在装载配置时解析范围

- 如果 `operator === "~"`，会拆出 `min/max`

影响：

- 媒体过滤规则表达力明显增强
- 但配置结构发生了兼容性升级，后续改版时不能再把 `Ext/Type` 当成旧结构

## 5. 本轮更新后的系统现状评估

### 5.1 变强的地方

- `m3u8` 工作流更完整，尤其是合并、内嵌 ffmpeg、切片列表交互
- `preview` 页面更像正式的管理台，不再只是简单预览
- 配置层表达力更强，尤其是大小过滤
- 文件命名、模板、DOM 提取能力更适合高级用户

### 5.2 变复杂的地方

- ffmpeg 调用链同时存在两套模式：
  - 远程标签页模式
  - iframe 模式
- m3u8 合并链已经跨越：
  - `popup.js`
  - `preview.js`
  - `function.js`
  - `m3u8.js`
  - `downloader.js`
- 状态来源进一步分散：
  - `G`
  - `cacheData`
  - `storage.sync`
  - `storage.session/local`
  - `sessionStorage`
  - 页面局部变量

### 5.3 当前最容易成为改版阻力的点

#### 1) `background.js` 过于中心化

它同时承担：

- 网络抓取
- 媒体过滤
- 缓存写入
- 脚本注入
- 消息总线
- tab 状态管理
- ffmpeg 转发

任何改动都容易产生连锁反应。

#### 2) `m3u8.js` 过于臃肿

它现在同时负责：

- URL 参数解析
- m3u8 解析
- ts 列表维护
- 密钥处理
- 下载逻辑
- ffmpeg 转发
- 多任务协同
- UI 渲染与交互

这是改版最值得拆的文件。

#### 3) popup 和 preview 存在重复逻辑

重复项包括：

- 批量下载
- m3u8 合并
- send2local
- 打开解析器
- 过滤/选中相关操作

后续功能如果双端都要加，会持续复制代码。

#### 4) 注入脚本天然高风险

尤其是 `search.js` 的 hook 范围很广，稍改就可能：

- 与站点脚本冲突
- 被站点检测
- 导致性能下降
- 引起兼容性回归

## 6. 改版建议：推荐拆分顺序

如果我们接下来要改一版，我建议按下面顺序推进。

### 第一阶段：先做结构性收敛，不动核心抓取算法

优先目标：

- 抽离“媒体动作层”
- 统一 popup / preview / downloader / m3u8 对资源对象的处理

建议抽成：

- `media-actions`
  - 下载
  - 合并
  - 发 aria2
  - 发 send2local
  - 发 MQTT
  - 打开 parser
- `media-model`
  - 标准化媒体对象字段
  - 文件名生成
  - 标题/扩展名兜底

### 第二阶段：拆 `background.js`

可以先按责任拆成：

- `background-capture`
- `background-filter`
- `background-message-router`
- `background-tab-state`

至少先把 `findMedia()` 和消息路由从一个大文件里拆开。

### 第三阶段：拆 `m3u8.js`

建议拆成：

- `m3u8-parser`
- `m3u8-download-controller`
- `m3u8-ffmpeg-bridge`
- `m3u8-ui`

这样以后无论是替换 UI，还是替换 ffmpeg 通道，都不会牵动整页逻辑。

### 第四阶段：最后处理注入脚本

只有在前面三步稳定后，再考虑：

- `search.js` 精简 hook 面
- 增加站点适配层
- 让 `catch.js` / `recorder.js` 有更清晰的生命周期管理

## 8. 本次结论

当前最新版本相比旧版本，最重要的增量不是“多了几个按钮”，而是三件事：

1. `m3u8` 流程升级成了更完整的解析/选择/合并/内嵌 ffmpeg 工作台
2. 配置系统开始支持更强表达力，尤其是资源大小过滤规则
3. `preview` 和下载链正在从简单页面，逐步演变成完整操作台

这对我们后续改版是好事，因为功能已经更完整；但也是风险，因为状态和逻辑变得更分散了。

如果下一步继续推进，最推荐的策略不是直接“边改边堆”，而是先做一轮模块收敛，至少把：

- 资源对象模型
- ffmpeg 桥接
- 批量下载动作
- parser 打开逻辑

这几块先抽平，再进入真正的功能改造。
